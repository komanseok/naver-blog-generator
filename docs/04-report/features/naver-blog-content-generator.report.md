### 4.1 다음 사이클로 이관

### 4.2 의도적 변경 사항

설계 문서에서 벗어난 변경사항들은 **모두 의도적**이며 **구현 품질 향상**을 목표로 한 것입니다.

#### 4.2.1 TipTap → contentEditable 에디터

**변경 사유**: TipTap v3는 Next.js 16 + React 19에서 SSR 호환성 문제 발생

**영향도**: 낮음 (에디터 기능 유지)

**상태**: ✅ MEMORY.md에 문서화됨

**권장사항**: TipTap이 SSR 이슈 해결될 때까지 현 구현 유지

---

#### 4.2.2 API 라우트 → Supabase 직접 호출

**설계**: 분리된 API 라우트 (GET /api/history, GET /api/history/[id], DELETE /api/history/[id])

**실제 구현**: Supabase 클라이언트 직접 호출 + localStorage 폴백

**변경 사유**: 더 간결하고 빠른 프로토타이핑, 오프라인 기능 지원

**영향도**: 낮음 (기능 동일, 약간 빠른 응답)

**권장사항**: MVP 단계에서는 현 구현 유지, 프로덕션 레이트 제한이 필요하면 API 라우트 추가

---

#### 4.2.3 Mock AI Provider 추가

**설계**: Claude, GPT, Gemini만 지원

**실제 구현**: Mock 제공자 추가 (테스트용)

**변경 사유**: API 키 없이 개발/테스트 가능

**영향도**: 긍정적 (개발자 경험 향상)

**권장사항**: 개발 환경에서 계속 유지

---

#### 4.2.4 AI 모델 버전 업데이트

**설계**: gpt-4o, gemini-2.0-flash

**실제 구현**: gpt-4.1-mini, gemini-2.5-flash

**변경 사유**: 최신 모델 (더 좋은 품질/비용 비율)

**영향도**: 긍정적 (더 나은 성능)

**권장사항**: 설계 문서 업데이트

---

#### 4.2.5 프롬프트 엔지니어링 강화

**설계**: 기본 SEO 규칙

**실제 구현**: 네이버 블로그 알고리즘 최적화 전략 포함된 프로덕션급 프롬프트

**변경 사유**: 실제 세계 최적화 (네이버 상위 노출 목표)

**영향도**: 높은 긍정적 영향 (콘텐츠 품질 향상)

**권장사항**: 설계 문서 업데이트

## 6. 배운 점과 회고

### 6.1 잘된 점 (Keep)

✅ **충분한 설계 문서의 가치**
- 설계 단계에서의 명확한 문서화가 구현 효율성을 크게 높였습니다.
- 단계별 명세가 개발자의 혼란을 줄이고 일관성 있는 구현을 가능하게 했습니다.

✅ **Feature 기반 모듈화**
- Feature 폴더 구조로 관심사의 분리가 명확했습니다.
- 각 기능이 독립적으로 테스트 가능하고 재사용 가능했습니다.

✅ **AI Provider 패턴**
- Provider 인터페이스를 통한 추상화로 새로운 AI 모델 추가가 간단합니다.
- 테스트를 위한 Mock Provider 추가도 용이했습니다.

✅ **Type Safety**
- TypeScript strict 모드로 런타임 오류를 사전에 방지했습니다.
- 인터페이스 정의가 API 계약을 명확하게 했습니다.

✅ **Supabase 통합**
- Auth, RLS, 실시간 기능이 내장되어 있어 개발 속도가 빨랐습니다.
- PostgreSQL 기반으로 복잡한 쿼리도 가능합니다.

### 6.2 개선할 점 (Problem)

⚠️ **초기 기술 선택의 재검토**
- TipTap v3 호환성 이슈는 초기 조사 부족이었습니다.
- 의존성 버전 검증을 더 엄격하게 해야 합니다.

⚠️ **환경 변수 템플릿 누락**
- `.env.example` 파일이 처음부터 있었으면 온보딩이 더 쉬웠을 것입니다.
- 설정 단계를 설계 문서에 더 상세히 포함해야 합니다.

⚠️ **프롬프트 엔지니어링 초기 과소평가**
- 네이버 SEO 규칙을 더 자세히 조사했으면 초기 프롬프트가 더 정교했을 것입니다.
- 사용자 피드백 루프가 필요합니다.

⚠️ **성능 테스트 계획 부족**
- SSE 스트리밍 성능 측정을 더 체계적으로 해야 합니다.
- 대용량 데이터 테스트가 필요합니다.

### 6.3 다음에 시도할 것 (Try)

💡 **TDD (Test-Driven Development) 도입**
- 핵심 로직(SEO 계산, 프롬프트)을 테스트 먼저 작성하면 품질이 높아질 것 같습니다.
- 현재는 수동 테스트에 의존하고 있습니다.

💡 **E2E 테스트 자동화**
- Playwright/Cypress로 전체 플로우를 자동 테스트하면 회귀 버그를 줄일 수 있습니다.
- 특히 AI 생성과 SEO 점수 계산을 검증하는 것이 중요합니다.

💡 **사용자 피드백 루프**
- Beta 테스터 그룹을 만들어 실제 네이버 블로그 최적화 효과를 측정해야 합니다.
- 프롬프트 개선이 계속 필요합니다.

💡 **분석 및 모니터링**
- Vercel Analytics, Sentry로 에러와 성능을 추적해야 합니다.
- 사용자의 가장 많이 사용하는 기능을 파악할 수 있습니다.

💡 **문서화 자동화**
- API 문서를 Swagger/OpenAPI로 자동 생성하면 유지보수가 쉬워집니다.

## 10. 개발자 공유 사항

### 10.1 주요 기술 결정

**TipTap 대신 contentEditable 사용**
- Next.js 16 + React 19 호환성 문제로 인해 변경
- 커스텀 툴바로 기본 기능 충분히 제공
- 프로덕션 코드: `src/components/editor/ContentEditor.tsx`

**Supabase 클라이언트 직접 호출**
- API 라우트 미들웨어 대신 클라이언트 SDK 직접 사용
- 더 빠른 프로토타입, 오프라인 지원 (localStorage)
- 향후 프로덕션 레이트 제한 필요 시 API 라우트 추가

**프롬프트 엔지니어링 강화**
- 네이버 블로그 알고리즘 최적화를 초기 설계보다 훨씬 깊이 있게 구현
- 코드: `src/lib/ai/prompts.ts`
- 문체, 길이, 키워드 자연 삽입이 모두 포함됨

### 10.2 코드 품질 하이라이트

✅ **Clean Architecture**
- 계층이 명확하게 분리됨 (Presentation → Application → Domain → Infrastructure)
- 의존성이 항상 한 방향 (위에서 아래로만)

✅ **Type Safety**
- TypeScript strict 모드 적용
- 모든 인터페이스가 명확하게 정의됨
- Union types로 AI Provider 확장성 확보

✅ **에러 처리**
- 모든 API 호출에 try-catch 및 사용자 친화적 에러 메시지
- Sonner 토스트로 사용자에게 피드백

✅ **성능**
- SSE 스트리밍으로 대기 시간 최소화
- 키워드 중복 캐싱으로 API 비용 절감
- 컴포넌트 지연 로딩 (next/dynamic)

### 10.3 설정 가이드 (개발자용)

**필수 환경 변수**:
```env
# AI API Keys (서버측)

# Supabase (클라이언트측)
NEXT_PUBLIC_SUPABASE_URL=https://xxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJ...

# Supabase Service Role (서버측)

```

